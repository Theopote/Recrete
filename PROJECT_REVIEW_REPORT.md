# Recrete 项目全面审查报告

## 执行摘要

Recrete 是一个针对既有建筑更新改造的 AI 设计助手平台，目前处于 MVP 试用阶段（第 6 个月）。经过全面审查，项目在**核心改造能力**方面已建立坚实基础，但在**某些关键业务场景**和**生产就绪性**方面存在需要改进的地方。

**总体评估：7.5/10** - 具备核心能力，可支持小规模试用，但需在特定领域强化才能胜任大规模生产应用。

---

## 一、项目概况

### 1.1 项目定位与目标
- **核心价值主张**：AI 驱动的建筑改造方案生成与诊断系统
- **目标用户**：建筑师、结构工程师、项目经理
- **关键差异化**：不是通用项目管理工具，而是专注于"理解建筑 → 诊断问题 → 生成可讨论的改造方案"

### 1.2 技术架构
- **前端**：Next.js 15 (App Router) + React 19 + TypeScript
- **样式**：Tailwind CSS + Radix UI
- **数据层**：Prisma + PostgreSQL（可选），带内存 mock 模式
- **状态管理**：Zustand + TanStack Query
- **AI 层**：Mock/OpenAI 双模式，支持场景化模型路由
- **代码规模**：约 497 个 TS/TSX 文件，18 个测试文件

### 1.3 核心功能模块
1. **AI 项目创建**：一句话描述生成完整项目
2. **Building Memory**：持久化建筑知识图谱
3. **文档分析**：PDF/DWG/图纸智能解析
4. **AI 诊断**：结构/MEP/消防/节能等多维度评估
5. **策略实验室**：生成 Light/Medium/Deep 三档改造方案
6. **合规检查**：基于规则引擎的 GB 规范检查
7. **成本估算**：基于案例库和区域基准的造价预测
8. **BIM 集成**：IFC/DWG 模型查看与空间标注

## 二、核心功能模块深度评估

### 2.1 ✅ 优势与亮点

#### 2.1.1 AI 能力架构设计优秀
- **多 Agent 架构**：18 个专业 Agent（结构、MEP、消防、节能等）
- **场景化路由**：Vision/Reasoning/Fast/Compliance 分场景选模型
- **Mock/Real 双模式**：无 API Key 也能演示，降低试用门槛
- **证据链设计**：`SourceEvidence` 模型追踪诊断/方案的来源

```typescript
// 设计亮点：evidence-driven diagnosis
export interface SourceEvidence {
  sourceType: 'document' | 'photo' | 'site_issue' | 'user_note';
  documentId?: string;
  pageNumber?: int;
  quote?: string;  // 原文引用
  boundingBox?: string;  // 图纸位置
}
```

#### 2.1.2 合规检查引擎专业
- **规则引擎**：70+ 条 GB 规范规则（防火、结构、无障碍、节能）
- **场景驱动**：根据项目类型自动筛选适用规范
- **诊断生成**：合规检查结果自动转换为 Diagnosis Item
- **气候分区**：支持中国建筑气候区划（严寒/寒冷/温和等）

#### 2.1.3 数据模型设计完善
- **Prisma Schema**：883 行，31 个模型，覆盖全业务流程
- **多租户隔离**：Organization 级别数据隔离
- **版本化设计**：策略评审、文档分析任务可追溯
- **枚举丰富**：RiskLevel/DiagnosisCategory/StrategyType 等

#### 2.1.4 知识库体系
- **案例库**：renovationCases 存储历史项目（成功/失败案例）
- **成本基准**：CostBenchmark 按区域/建筑类型/策略类型聚合
- **材料价格**：MaterialPrice 按地区跟踪材料价格指数
- **规范库**：buildingCodes 结构化存储 GB 规范条文（714 行代码量）

#### 2.1.5 Building Memory 设计创新
这是 Recrete 的核心品牌资产，持久化建筑知识：
- Known Facts（AI 已知）
- Missing Information（AI 未知）
- Key Risks（关键风险）
- Renovation Potential（改造潜力）
- Design Constraints（设计约束）

### 2.2 ⚠️ 存在的问题与风险

#### 2.2.1 业务场景覆盖不足

**问题 1：复杂结构加固场景处理简单**
```typescript
// lib/ai/agents/structural-agent.ts
suggestStrengtheningMethods(issue: 'insufficient_capacity' | 'corrosion' | 'seismic' | 'crack')
```
- 仅支持 4 类加固场景，实际老旧建筑改造会遇到：
  - 砌体结构抗震加固（增设钢筋混凝土框架）
  - 地基不均匀沉降处理
  - 混凝土梁柱节点加固
  - 预制板结构改造
  - 木结构保护性加固

**问题 2：MEP 系统诊断缺失**
- 存在 `mep-agent.ts`，但功能未实现（仅空壳）
- 老旧建筑改造中，MEP 系统（给排水、暖通、电气、弱电）是重点：
  - 管线冲突检测
  - 设备机房布局优化
  - 电气容量复核
  - 消防系统升级方案

**问题 3：遗产建筑保护深度不够**
- Schema 有 `HeritageLevel` 枚举，但缺少：
  - 保护建筑修缮导则库
  - 历史价值评估框架
  - 原真性保护策略
  - 可逆性设计原则

**问题 4：成本估算精度有限**
```typescript
// lib/ai/agents/cost-estimator-agent.ts
const baselineFromCases = references.length > 0
  ? references.reduce((s, c) => s + (c.costPerSqm ?? 2500), 0) / references.length
  : benchmark?.costPerSqmAvg ?? (strategyType === 'light_renewal' ? 1200 : 4200);
```
- 估算依赖案例库和区域基准，但：
  - 案例库规模未知（需查看 mock-data）
  - 缺少 WBS（工作分解结构）详细分项
  - 未考虑地方定额差异
  - 缺少动态材料价格联动

#### 2.2.2 数据持久化不完整

根据 `docs/trial-known-issues.md`：
- ❌ `updateBuildingMemory` 在 DB 模式下仍委托 mock（第 6 个月已修复）
- ❌ `addStrategyVersion` 方案版本历史仅 mock-only
- ❌ 完整审计日志未实现

#### 2.2.3 测试覆盖不足

**测试现状**：
- 18 个测试文件（单元测试为主）
- `npm run test` 因 rollup 依赖问题无法运行（VM 环境）
- 缺少 E2E 测试（Playwright）
- smoke 测试只有 5 项核心检查

**关键业务路径未覆盖**：
- ✅ 有：core-journey.test.ts（核心流程）
- ✅ 有：compliance-store.test.ts（合规存储）
- ❌ 缺：文档上传 → 分析 → 诊断生成全流程
- ❌ 缺：策略生成 → 迭代 → 对比全流程
- ❌ 缺：Building Memory 更新完整性

#### 2.2.4 UI/UX 实现问题

**中英混排严重**
- 组件标题中英文混杂，如 "Strategy Lab · AI 策略实验室"
- 缺少统一的国际化方案（i18n）

**移动端支持不足**
- 主要为桌面浏览器设计
- 现场勘查场景无法使用移动设备录入

**文档体验需优化**
- 缺少文档预览器（PDF inline 预览）
- 文档分析进度不透明
- 缺少批量操作（批量删除/分类）

#### 2.2.5 生产部署风险

**已知问题（P0-P2）**：
1. **P0**：Vercel + 本地存储 → 未配置 S3/R2 时文件不持久
2. **P1**：数据库迁移 → 新模型需手动 `db:push`
3. **P1**：无 OpenAI Key → 无法评估真实 AI 质量
4. **P2**：大 PDF/DWG 分析超时（多页 PDF 限 5 页）
5. **P2**：后台任务无 Redis 时单实例派发（无水平扩展）

**RBAC 不完整**：
- 基础角色权限存在，但缺少：
  - 组织成员管理 UI
  - 自助邀请同事流程
  - 审计日志
  - 细粒度资源权限（项目级/文档级）

### 能胜任的场景 ✅
1. **中小型办公楼改造**（3000-10000㎡）
2. **功能置换项目**（办公 → 文化/商业）
3. **初步方案阶段**（前期诊断 + 概念方案）
4. **合规性预审**（规范条文快速检查）

### 存在困难的场景 ⚠️
1. **大型复杂项目**（>20000㎡，多栋，分期）
2. **遗产建筑修缮**（历史价值评估需人工）
3. **深化设计阶段**（详细 BIM 协同、施工图）
4. **特殊结构加固**（如砌体抗震、预制板改造）
5. **MEP 系统复杂改造**（如数据中心、医疗建筑）

### 无法胜任的场景 ❌
1. **无图纸老旧建筑**（依赖现场测绘，系统无 3D 重建）
2. **抗震鉴定加固**（需专业软件如 PKPM）
3. **施工过程管理**（非项目管理工具）
4. **成本精算**（缺少完整定额库）

## 五、风险与缓解措施

### 5.1 技术债务风险 ⚠️

**风险**：
- Mock 模式下部分功能不持久（已知）
- 测试覆盖不足，重构易出错
- 单实例架构，无水平扩展

**缓解**：
- ✅ 已有 `docs/trial-known-issues.md` 透明告知
- 建议：第 2 家试用前清空 P0/P1 backlog
- 建议：≥3 家用户时上 Redis + BullMQ

### 5.2 AI 质量风险 🔴

**风险**：
- Mock 模式下无法评估真实 AI 效果
- LLM 幻觉可能导致错误诊断/方案
- 缺少专业审核机制

**缓解**：
- **强制要求**：试用必须配置 OpenAI Key
- 建议：所有 AI 输出标注 `confidence` 分数
- 建议：关键诊断（结构/消防）强制标记 `requiresEngineerReview: true`
- 建议：增加"人工审核"工作流节点

### 5.3 数据安全风险 🔴

**风险**：
- 租户隔离未经充分测试
- 敏感图纸上传到第三方 AI API
- 缺少审计日志

**缓解**：
- ✅ 已有 `tenant-isolation.test.ts`
- 建议：增加 E2E 租户隔离测试
- 建议：文档上传时明确隐私政策
- 建议：支持私有化部署（自建 LLM）

### 5.4 业务风险 ⚠️

**风险**：
- 方案质量依赖 prompt 工程
- 缺少行业专家验证
- 用户过度依赖 AI 判断

**缓解**：
- 建议：与结构工程师/建筑师深度合作优化 prompt
- 建议：所有方案页面加免责声明："AI 生成方案仅供参考，需专业人员审核"
- 建议：建立专家反馈循环（feedback-triage-workflow.md）

## 六、最终结论与建议

### 6.1 核心判断

**Recrete 当前状态：成熟的 MVP，可支持小规模试用**

✅ **优势**：
1. AI 架构设计专业，多 Agent + 场景路由
2. 合规检查引擎达到生产级别
3. 数据模型设计完善，扩展性强
4. Building Memory 创新设计有品牌价值

⚠️ **不足**：
1. 部分业务场景覆盖不足（MEP/遗产/复杂结构）
2. 成本估算精度有限
3. 测试覆盖不足，生产稳定性待验证
4. AI 质量依赖 Key 配置和 prompt 优化

### 6.2 能否胜任老旧建筑改造？

**回答：部分胜任，需明确边界**

**适合场景**（推荐使用）：
- 中小型商业办公楼改造（<10000㎡）
- 功能置换项目（办公 → 文化/商业）
- 前期阶段（诊断 + 概念方案）
- 快速合规性预审

**不适合场景**（需人工主导）：
- 遗产建筑修缮（历史价值评估）
- 复杂结构加固（抗震/地基）
- 深化设计与施工图阶段
- 大型多栋分期项目

### 6.3 行动建议

#### 对于项目团队：

**短期（1-2 个月）**：
1. 清空 P0/P1 已知问题
2. 配置生产环境（S3 + PostgreSQL + OpenAI）
3. 完成第一家试用，收集真实反馈
4. 实现 MEP Agent 基础功能

2. 优化 UI/UX（国际化 + 移动端）
3. 实现 BIM 深度集成
4. 建立专家审核工作流
5. 准备第二家、第三家试用

**长期（6-12 个月）**：
1. 遗产建筑保护模块
2. 私有化部署支持
3. 行业案例库扩充（100+ 案例）

#### 对于潜在用户：

**适合你的情况**：
- 你有 **3-5 个**中小型改造项目在手
- 你需要快速生成**初步方案**给甲方
- 你希望**自动化合规检查**节省时间
- 你的团队**愿意尝试 AI 辅助设计**

**慎重考虑**：
- 你主要做遗产建筑修缮（系统不成熟）

- 你的项目涉及复杂结构加固（需专业软件）
