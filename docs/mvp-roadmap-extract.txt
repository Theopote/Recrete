六个月独立开发路线图 · 面向事务所试用的MVP
前提：单人开发 · 非程序员背景，借助AI编程工具 · 目标是6个月内让1-2家建筑事务所能真实试用
0. 先调整判断标准
上一版任务书是按“生产级产品”的标准写的，对你现在的情况不合适。给事务所试用的MVP和正式商业化运营，能接受的粗糙程度完全不同。这一版按你的真实目标重新排了优先级，原则是：
能让1-2家事务所放心把真实项目数据放进来试用 —— 这是唯一的硬指标
完整的测试体系、真任务队列、微服务拆分 —— 这些留到试用后有真实反馈、且用户量真的增长时再做，现在做是浪费时间
你是建筑师不是程序员，所以路线图里我会尽量说“要做成什么效果”和“为什么重要”，而不是甩术语，具体怎么改交给AI编程工具（Cursor/Claude Code）去做，你负责验收结果对不对
1. 本次复查中发现的紧急问题
在上一份评估之后，我针对性检查了“不同事务所的数据会不会串”这件事，结果如下，这个问题的优先级要放在所有其他工作之前：
严重问题：项目、文档、建筑记忆、策略方案等几乎所有接口，都只检查“你有没有登录”，没有检查”这个项目是不是你所在事务所的”。现在只要是登录用户，换一个项目ID的网址就能看到任何其他事务所的建筑数据。如果两家事务所同时试用，这是会立刻出事的问题，不是理论风险。
好消息是：数据库模型里已经有 Organization（事务所/组织）和 User 的关联设计，缺的只是“每次查询数据时，顺手带上‘这是谁在问、只给他能看的’这一层过滤”。这是一个模式统一、可以让AI编程工具批量处理的修复，不是要推倒重来。

3. 分月详细任务
第1个月 · 数据安全与可部署（最优先，不可跳过）
修复数据隔离问题：所有按项目查询的接口都要先确认“当前登录用户所属事务所 = 该项目所属事务所”，不满足就拒绝访问。这一条建议明确写给AI编程工具，要求它“扫描所有 app/api/projects 下的路由，统一加上这层校验”，然后你重点验收：用两个不同账号互相试着访问对方项目的网址，看是否被拒绝。
把 USE_DATABASE=true 的数据库模式跑通并作为唯一正式模式：内存模式适合你自己演示，但事务所试用必须是真实存库，否则重启一次数据就没了。
找一个能一直开着的地方部署（推荐 Vercel + 托管版 PostgreSQL，如 Supabase/Neon，配置比自己租服务器简单很多），拿到一个事务所随时能打开的网址。
修复安装环境问题（镜像源地址、Next.js版本），确保换一台电脑也能一次装好，不然以后你自己都可能卡住。
验收标准：两个测试账号分属两个虚拟事务所，互相看不到对方项目；关掉电脑重开，网址依然能访问、数据还在。
第2个月 · 让AI能力是真的
接入真实的 OpenAI 或 Anthropic API Key，替换默认的 Mock 服务 —— 目前所有“AI生成方案/AI诊断”默认是预设的假数据，事务所试用時必须是真实调用。
处理AI调用失败的情况：网络问题、超时、API报错时，界面要显示“正在重试/请稍后”而不是白屏或报错堆栈，这一点对非技术用户的第一印象影响很大。
做一个简单的用量控制：比如给每个事务所每天/每月设一个AI调用次数上限，避免因为某次误操作产生意外的大额账单（这一点很多独立开发者会忽略，等账单来了才后悔）。
验收标准：断网重连后AI功能能自动恢复；连续用一天，AI调用产生的费用在你能接受的范围内并且可以查到明细。
第3个月 · 账号与权限收尾
把现在写死的三个演示账号，换成事务所自己能用的邀请注册流程：你给事务所负责人发一个邀请链接，他自己设密码，再邀请同事加入同一个事务所空间。
基础密码重置（忘记密码能自己找回，不用每次找你）。
不需要做复杂的角色权限（建筑师/结构/甲方分权限），先做到“同一事务所内部互相能看，不同事务所互相看不到”就够用，这是第1个月已经打好的地基，这里是补上注册和邀请环节。
验收标准：你自己不用手动在数据库里建账号，事务所负责人可以自助邀请2-3个同事进来。
第4个月 · 核心体验打磨
重点打磨三个此前讨论过、最能体现产品价值的环节：一句话创建项目的流式反馈是否流畅、AI生成的改造方案是否清楚易懂、方案对比是否能让建筑师快速做出判断——这三步是事务所第一次用的“前3分钟”，决定他们会不会继续用下去。
协作评审功能目前完全空白，第4个月先做最简版：允许在方案页面留言/@提到某个具体风险点即可，不需要做成实时多人同屏协作，那个复杂度对单人开发不现实，先满足“甲方和结构工程师能在同一个页面上留言讨论”这个最基本需求。
给关键功能补一层轻量的自动检查（不是完整测试体系，就是5-8个“每次改完代码，跑一下这几步确认没崩”的自动化脚本，覆盖登录、建项目、AI生成方案、数据隔离这几个最要命的环节）。这层薄薄的安全网对你格外重要——因为你不是程序员，没法靠读代码判断AI改动有没有引入新问题，跑一下脚本比你自己一页页点更可靠。
验收标准：让一个完全没见过这个产品的建筑师朋友，独立完成“注册→建一个项目→生成一版改造方案”，全程不用你在旁边解释。
第5个月 · 小范围真实试用
找1家你熟悉、愿意包容Bug的事务所，让他们拿一个真实（或接近真实）的项目试用，而不是找完全陌生的人——第一轮试用的目的是收集问题，不是验证市场。
你全程盯着，重点记录：他们在哪一步卡住了、哪句话看不懂、AI给出的方案是否真的有参考价值。这些反馈比你自己闭门造车判断准得多。
这个月不建议再加新功能，除非试用中发现某个环节完全用不下去。
第6个月 · 根据反馈收口
集中修复第5个月暴露出的真实问题，这些问题的优先级天然高于你自己想加的新功能。
整理一份给第二家事务所看的简单介绍材料（可以用我们之前讨论的几个界面设计作为素材）。
回头看一眼上一版任务书里“阶段二/三/四”的内容（真队列、完整测试体系、正式RBAC等），根据试用后的真实用户量决定要不要启动，大概率这个阶段还不需要。
4. 给非程序员单人开发者的几条建议
你现在已经用AI编程工具做出了一个功能相当完整的应用，这本身说明这条路径是可行的，不用因为不会写代码而缺乏信心。
每次让AI编程工具改动代码时，尽量一次只解决一个具体问题（比如“只修数据隔离”），改完你验收一次，而不是一次性提一堆需求——单人非程序员最大的风险是攒了一堆改动后不知道哪个环节出的问题。
第4个月建议的“轻量自动检查脚本”值得优先做，这是你在不会读代码的情况下，最有效的“有没有搞坏”的判断工具。
涉及数据安全（第1个月的问题）这类内容，建议改完之后明确要求AI编程工具解释一遍“改了什么、为什么这样能解决问题”，即使你不懂具体代码，也能通过它的解释判断思路是否合理。
